# 每日通知模板:緊湊化 + 步行資訊設計

日期:2026-06-27

## 背景與問題

`templates/daily-notify-template.md` 是每日房源通知的輸出模板。使用者提出兩個訴求:

1. **加上最近捷運的步行資訊**:步行時間、是哪個捷運站、幾號出口,並附一個可直接
   打開該物件位址的地圖連結。
2. **單筆物件太長**:目前一筆推薦物件有 14 行欄位,讀起來太長。

## 目標(已與使用者確認的決策)

1. **精簡策略 = 合併不刪**:欄位資訊原則上保留,把多行擠成少數幾行;唯一移除的兩
   個欄位是「狀態」(區塊標題已表明)與「刊登日」(推薦/接近門檻看的是數字)。
2. **緊湊程度分級(Q1=C)**:推薦/接近門檻用「完整緊湊版」;前置排除/可疑/目標日
   排除用「更短版」。
3. **步行行範圍(Q2=C)**:🚶 步行行放進 **推薦、接近門檻、前置排除** 三類(前置排除
   的原因常常就是離捷運太遠,放了剛好佐證);可疑/待查與目標日排除不放。
4. **資料來源不變**:步行資訊與座標都已存在於 enriched 資料,本設計只改模板與
   `reporting-rules.md` 的呈現準則,**不動任何抓取/enrich 程式碼**。

## 非目標(YAGNI)

- 不改快速摘要區(兩個訴求都是針對單筆物件,摘要維持原樣)。
- 不新增任何 enrich 欄位、不改 `scripts/` 下的程式邏輯。
- 不改投資門檻、排序、排除判斷等規則。

## 資料來源(已存在,無需新增)

每筆 `EnrichedListing`(見 `scripts/lib/types.ts`)已具備呈現所需的全部欄位:

- `walk`:`{ stationZh, line, exitId, distanceM, minutes }` —— 站名、出口、步行
  分鐘、routed 公尺。**只有資料可信時才非 null。**
- `coordinate`:`{ lat, lng }` —— 用來組地圖連結。
- `withinWalk` / `reliability.reason` —— 判斷步行行要走哪個資料狀態。
- `addressOrArea`、`url`、各數值欄位、`signals.auctionKeyword`。

## 步行行(🚶)呈現規則

地圖連結格式:`https://www.google.com/maps?q=<lat>,<lng>`,連結文字為「地圖」。
依資料狀態三選一(由 agent 依 `reliability` 判定):

1. **資料可信**(`walk` 非 null):
   `🚶 {stationZh} {exitId} 號出口・{minutes} 分鐘（[地圖](…)）`
   - 出口資料缺漏時退化為 `🚶 {stationZh}・{minutes} 分鐘（[地圖](…)）`。

2. **不可信但有座標**(`walk` 為 null、`coordinate` 非 null,如座標與地址不一致、
   繞路離譜):顯示 agent triage 結果,標明待確認 ——
   `🚶 約{triage 推測站}・步行待確認（[地圖](…)）`
   - triage 推不出站別時:`🚶 步行待人工確認（[地圖](…)）`。

3. **完全沒座標**(`coordinate` 為 null,連地圖都生不出來):
   `🚶 無位置資訊`(不附連結)。

## 各區塊版型

移除的共通欄位:每筆不再有 `- 狀態：…` 行(區塊標題已表明);推薦/接近門檻不再有
`- 刊登日：…`。連結沿用既有規則:標題一律 Markdown 連結到 `url`。

### ① 推薦(完整緊湊 + 步行)

```
#### 1. [美寓](url) ｜ 低於行情 11%・覆蓋率 1.09
- 🚶 東門站 4 號出口・8 分鐘（[地圖](https://www.google.com/maps?q=25.03,121.52)）
- 1000 萬／20 坪／50 萬/坪・3/5 樓・屋齡 30・中正區X街
- 行情 56 萬/坪・月租 ~35000・房貸 32031・現金流 +2969/月
- 推薦理由：…
- 風險：…
```

標題行帶「低估幅度・覆蓋率」兩個關鍵指標。四條細節行:步行、物件基本盤、財務、
理由、風險。

### ② 接近門檻(同推薦,標題改放覆蓋率 + 差一點原因)

```
#### 1. [華廈](url) ｜ 覆蓋率 0.85・差在低估幅度 7%
- 🚶 大安站 2 號出口・10 分鐘（[地圖](…)）
- 1200 萬／18 坪／67 萬/坪・5/12 樓・屋齡 15・大安區Y路
- 行情 72 萬/坪・月租 ~38000・房貸 38000・現金流 0/月
- 需人工確認：…
```

### ③ 前置排除(更短 + 步行)

```
#### 1. [公寓](url)
- 🚶 東門站 4 號出口・14 分鐘（[地圖](…)）
- 前置排除：離捷運 >800m（routed 1100m）
```

步行行先擺,因為前置排除原因常常就是它;第二行為排除原因 + 證據合併。

### ④ 可疑/待查(更短,無步行)

```
#### 1. [套房](url) ｜ `likely-auction`
- 命中訊號：法拍關鍵字、無室內圖
- 理由：…（信心：中・已點詳情頁）
```

標題行帶 `clean`/`suspicious`/`likely-auction` 標籤。

### ⑤ 目標日排除(更短,無步行)

```
#### 1. [大樓](url)
- 1000 萬／20 坪／50 萬/坪・覆蓋率 0.6
- 排除：租金覆蓋率不足
```

## 模板實作要點

模板由 agent 讀 enriched 資料後填寫。新增/調整的 placeholder 概念:

- 標題行:推薦/接近門檻把關鍵指標併入 `####` 標題行;可疑類把標籤併入。
- `{{walk_line}}`:由 agent 依上節三狀態規則組好整行字串(含地圖連結或「無位置
  資訊」),放進推薦/接近門檻/前置排除三區塊。
- 細節行用全形間隔點「・」分隔同一行的多個欄位,斜線「／」分隔總價/坪數/單價。
- 各區塊的 `{{else}}`(無資料)文案沿用現有版本。

## reporting-rules 改動

在 `docs/reporting-rules.md` 的「Notification Format」一節補上:

- 步行行(🚶)三狀態的組字規則與地圖連結格式(`maps?q=lat,lng`)。
- 每筆移除「狀態」行;推薦/接近門檻移除「刊登日」。
- 緊湊分級:推薦/接近門檻完整緊湊版、前置排除/可疑/排除更短版。

其餘排序、字數上限(單則約 3,500 中文字)、連結等規則不變,且緊湊化讓字數更有餘裕。

## 邊界 / 錯誤處理

- `walk` 與 `coordinate` 皆缺:步行行走狀態 3(`🚶 無位置資訊`)。
- 出口 `exitId` 缺但有站名/分鐘:退化顯示,不顯示「號出口」。
- 數值欄位(月租、現金流等)為 null:該欄以 `—` 帶過,不讓整行崩掉。
- 地圖連結只在有 `coordinate` 時生成;狀態 3 不附連結。

## 影響檔案清單

- `templates/daily-notify-template.md`(改:5 區塊版型 + 步行行 + 移除欄位)
- `docs/reporting-rules.md`(改:Notification Format 補步行行/地圖/緊湊規則)

無程式碼(`scripts/`)改動,因此無新增自動化測試;以模板與規則文件作為規格,靠人工
檢視日報驗證。
