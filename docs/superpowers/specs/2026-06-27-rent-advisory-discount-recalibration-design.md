# 租金降為參考 + 開價溢價門檻重構（投資 profile）

- 狀態：設計待審
- 日期：2026-06-27
- 範圍：investment profile 的篩選邏輯與通知呈現
- 性質：純文件 + 一張參考表；零程式邏輯改動、零新測試

## 背景與問題

investment profile 原本用兩根支柱篩選物件：

1. **折扣**：`低於行情 >= 10%`，行情錨定實價登錄（成交價）。
2. **租金覆蓋率**：`租金覆蓋率 >= 1.0`（推薦）／`>= 0.8`（接近門檻），其中
   `租金覆蓋率 = 預估月租 / 月房貸`。

審視後發現兩個結構性缺陷：

### 缺陷 A：租金是 pipeline 裡最不可靠的輸入，卻有否決權

- 租金完全靠 agent 臨場估算，**無資料源、無快取、無新鮮度規則、不可重現**
  （對比走路距離有 ORS + MRT CSV + cache，行情有實價登錄錨點）。
- 一個 ±20% 都可能的點估計，被當成 `覆蓋率 >= 1.0` 的二元硬門檻 →
  物件在門檻兩邊隨雜訊跳動。**用拿不準的數字做決定性判斷**。

### 缺陷 B：折扣基準有系統性偏差（拿開價比成交價）

- 折扣公式 `(行情 − 物件單價) / 行情` 中，行情是**成交價**等級（實價登錄），
  物件單價卻是 iBigFun 上的**開價**（委託價），系統性偏高。
- 結果：幾乎每筆都算成「高於行情」（負折扣），`低於行情 >= 10%` 這個門檻在
  此基準下幾乎永不觸發；真正算出 `>= 10%` 的反而多是法拍/異常。**門檻壞了。**

## 設計決策（已與使用者確認）

1. **租金降為純參考**：只顯示、不影響分桶。不建實價登錄租金資料集；agent 粗估即可。
2. **指標改為「開價溢價」**：把折扣翻成溢價（開價相對成交行情高多少），沿溢價光譜重畫門檻。
3. **門檻錨定各市議價率**：用永慶房屋每季公布的成交議價率換算的典型溢價 `p*`。
4. **推薦採相對門檻**：`溢價 ≤ p*/2`（開價溢價明顯低於該市典型即推薦）。
5. **議價率以參考表維護**：每市一列、手動每季更新，比照 MRT CSV 的維護模式。

被明確排除的選項：完全移除租金（保留量級訊號）、爬 591 即時租金（違背可重現）、
建實價登錄租金資料集（租金降為參考後不需要）、改動覆蓋率指標本身（改顯示現金流）。

## 指標定義

```
開價溢價 (premium) = (物件開價單價 − 成交行情單價) / 成交行情單價      # 越低越好；可為負
議價率 (r)         = 該市成交議價率 = (開價 − 成交) / 開價            # 永慶季報定義
典型溢價 (p*)      = r / (1 − r)                                      # 換算成「相對成交」的溢價
```

- 行情（成交行情單價）的取得**沿用現行實價登錄優先序**（實價登錄連結 → agent 可比成交 →
  弱資料），不變。溢價建立在這個既有的行情估算之上。
- 議價率分母是開價、溢價分母是成交，兩者不同；參考表須記錄此換算。

## 分桶門檻

| 分桶 | 門檻 | 台北示例（r=12.8% → p*≈14.7%） |
|---|---|---|
| **推薦** | `−10% < 溢價 ≤ p*/2` 且 走路可靠在內 且 乾淨 且 行情資料可靠不過期 | 溢價 ≤ ~7.3% |
| **接近門檻** | `p*/2 < 溢價 ≤ p*`，或溢價達推薦級但只差在行情待確認／走路待確認 | ~7.3% ~ ~14.7% |
| **排除** | `溢價 > p*` | > ~14.7% |
| **可疑/待查** | `溢價 ≤ −10%`（異常低，先驗證）＋ 既有可疑訊號（法拍/低資訊等） | 開價遠低於成交 |
| **前置排除** | 可靠步行 > 10 分（投資 profile 硬排除）— **不變** | — |

規則細節：

- 推薦是**相對**門檻：`p*` 取物件所在市的議價率換算值；不同市的推薦線不同。
- 溢價 `≤ −10%` 的異常低物件先進可疑/待查驗證，**不直接推薦**（多為法拍/資料異常）。
  驗證為乾淨且行情可靠後，可依其溢價歸入推薦/接近。
- 行情資料若過期/弱/跨站，物件**不可標推薦**，降到接近門檻並標人工確認（沿用既有資料品質規則）。
- 走路 `withinWalk === false`（可靠且 >10 分）仍前置排除；`withinWalk === null` 不排除，
  走 triage（不變）。

## 排序

- 推薦、接近門檻、排除三桶一律按 **開價溢價由低到高**（溢價越低越前），次鍵總價低者優先。
- 取代原本「折扣由高到低 / 覆蓋率由高到低」的排序。

## 租金（純參考）

- 報表顯示 `月租 ~{{estimated_rent}}（參考·低信心）` 與
  `現金流 ~{{monthly_cash_flow}}/月（參考）`（現金流 = 月租 − 房貸）。
- **拿掉覆蓋率**（`{{rent_coverage}}`）：既不卡關、又把融資假設混進篩選。
- 租金由 agent 粗估，標來源（若有）與低信心；**永不影響分桶或排序**。
- `scripts/lib/finance.ts` 的 `rentCoverage()` 保留為工具函式，但註解標明僅供參考顯示、
  不再用於門檻判斷。既有測試保留。

## 議價率參考表（新增）

- 新檔 `data/negotiation-rate.md`：每市一列。

  欄位：`city`（縣市名）、`rate`（成交議價率 r，%）、`source_quarter`（如 `2025Q1`）、
  `source_url`。檔頭記錄：`p* = r / (1 − r)`、永慶議價率分母為開價、每季更新流程。

- 初始種子（[永慶 2025 Q1](https://knowhow.yungching.com.tw/article/4380)）：
  台北 12.8 / 新北 11.5 / 桃園 10.1 / 新竹 9.6 / 台中 9.3 / 台南 11.1 / 高雄 10.7。
- 維護：每季永慶季報公布後手動更新 `rate` 與 `source_quarter`，比照 `taipei_mrt_exits.csv`
  的手動刷新模式。表中無對應市別時，agent 退而使用最接近的都會區或全國值，並於該物件註記。
- `data/README.md` 新增本表的說明段落。

## 不確定性與新鮮度處理

- 議價率帶 `source_quarter`，過舊（如超過 2 季）時於報表快速摘要標註「議價率資料偏舊」。
- 議價率隨景氣波動（永慶資料顯示熱市 7–10%、冷市 11–13%），故參數須定期校準，不寫死於程式。
- 溢價落在門檻邊界（如接近 `p*/2`）時，agent 在該物件註記其對行情估算的信心，避免邊界誤判。
